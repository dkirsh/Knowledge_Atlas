## S1 — XSS via `innerHTML` interpolation 🟢 NOT VULNERABLE

**Risk model.** The result panel renders fields that originate from user uploads or PDF text content: `filename`, `citation`, `classifier_article_type`, `primary_topic`, `reason`, `routing_reason`, `duplicate_of`. A malicious PDF whose extracted title contains a string like `<img src=x onerror=alert(1)>` would, if inserted via `innerHTML`, execute attacker-controlled JavaScript in the reviewer's browser.

**Code review.** The result-card renderer at `ka_contribute_public.html:235-296` builds the DOM entirely via `document.createElement(...)` and `textContent` assignment. There is no `innerHTML =` assignment anywhere in the file. Verbatim source comment at lines 234-238:

```js
// Render result cards into the inline panel. Uses DOM construction with
// textContent (no innerHTML) so classifier-supplied strings cannot inject
// markup. APPENDS to prior content so successive submissions in one
// session show a running history.
```

**Verification.** `grep -nE "innerHTML" ka_contribute_public.html` returns **zero matches**. Every dynamic value passes through one of:
- `el.textContent = userValue` (line 248, 255, 270, 277, etc.)
- `document.createTextNode(userValue)` (line 272)
- `el.setAttribute("data-idx", String(i))` (line 275)

None of these execute markup.

**Status:** 🟢 **NOT VULNERABLE.** The XSS attack surface that exists in peers' code (where `innerHTML` is used in places with `escapeHtml()` discipline) does not exist in ours — there is no code path that can be made to execute markup, period. This is a structural safety property, not a discipline-dependent one.

## S2 — Path traversal via uploaded filename 🟢 NOT VULNERABLE

**Risk model.** `file.filename` is user-supplied. A payload like `../../etc/passwd` would, if used to construct a filesystem path, write outside the quarantine directory.

**Code review.** Quarantine path construction at `ka_article_endpoints.py:981`:

```python
quarantine_path = month_dir / f"{article_id}.pdf"
```

The path component is **always** `{article_id}.pdf`, where `article_id` is generated by `_next_id` (which returns strings of the form `KA-ART-NNNNNN` — only uppercase letters, digits, hyphen; no slashes possible). The user-supplied `file.filename` is stored in the `pdf_filename` database column for display/audit purposes only; it is never passed to any filesystem call.

The same pattern is used at line 2285 (the citation-only quarantine path).

**Status:** 🟢 **NOT VULNERABLE.** Path traversal is structurally impossible because the user does not control the filename component of the storage path.

## S3 — File magic-byte and size validation 🟢 ENFORCED

**Magic bytes.** `_validate_pdf_bytes` at `ka_article_endpoints.py:379-388`:

```python
if not data[:5] == b"%PDF-":
    notes["magic_bytes"] = "FAIL"
    notes["rejection_reason"] = "Not a PDF file (invalid magic bytes)"
    return notes
notes["magic_bytes"] = "PASS"
```

A file whose first 5 bytes are not exactly `%PDF-` is rejected with `status=rejected_bad_file`. No bytes are written to the quarantine directory.

**File size.** `MAX_FILE_SIZE_BYTES` at line 166 (default 100 MB, env-overridable via `KA_MAX_FILE_SIZE_MB`); checked at line 391-393. **Batch size:** `MAX_BATCH_SIZE_BYTES` at line 167 (default 500 MB); checked at line 811-813 and returns HTTP 413 if exceeded.

**Verification.** Contract TC-3 was run on 2026-05-19 against the live endpoint and produced `status=rejected_bad_file` with reason `"Not a PDF file (invalid magic bytes)"`. See `validation_TC3_response.json`.

**Status:** 🟢 **ENFORCED** at the gateway. Tested end-to-end.

## S4 — DoS via large upload buffered into memory 🟡 PARTIAL MITIGATION

**Risk model.** `await file.read()` (line 818 inside the per-file loop) reads the upload into memory before `_validate_pdf_bytes` runs. A 10 GB upload would buffer 10 GB before the size check returns failure, potentially crashing the worker on memory pressure.

**Mitigation in place.** FastAPI's `UploadFile` writes large request bodies to a `SpooledTemporaryFile` on disk (default spool threshold ~1 MB); once exceeded, memory pressure is bounded. The server-side check at line 811-813 enforces `MAX_BATCH_SIZE_BYTES` (500 MB default) and returns HTTP 413 before any further processing.

**Residual risk.** No starlette/Uvicorn-layer `--limit-max-body` is set in this PR. A malicious uploader could in principle stream up to the SpooledTemporaryFile capacity before the explicit check fires.

**Recommendation for production deployment.** Add a starlette middleware (or an Uvicorn `--limit-max-body` flag) that rejects request bodies over `MAX_BATCH_SIZE_BYTES + 1 MB` before the route handler is invoked. Not blocking for course submission.

**Status:** 🟡 **PARTIAL MITIGATION.** Documented; not blocking.

## S5 — SQL injection 🟢 NOT VULNERABLE

**Risk model.** Any `conn.execute(...)` call that uses string interpolation of user-controlled values into the SQL string is a potential SQL injection.

**Code review — submit_articles path.** Every INSERT in the submit-articles flow uses `?` parameter binding:
- `ka_article_endpoints.py:986-1027` — main `articles` INSERT with 30 columns, all `?` placeholders
- `ka_article_endpoints.py:1107-1140` — duplicate-existing INSERT, all `?` placeholders
- `ka_article_endpoints.py:1163-1182` — `audit_log` INSERT
- `ka_article_endpoints.py:732-735` — `submission_batches` INSERT

**f-string SQL in this module is structural, not data.** A grep for `f"SELECT` / `f"INSERT` returns three matches (lines 1270, 1272, 1582) — all build the *query shape* (which columns to project, IN-clause placeholder count), with user-supplied *values* still passed via `?`. Example at line 1582:

```python
f"SELECT * FROM research_questions WHERE question_id IN ({placeholders}) ORDER BY question_id"
```
where `placeholders = ",".join("?" for _ in question_ids)`. The interpolated `placeholders` is `?,?,?` — not user content. The actual `question_ids` are bound via `?`.

**Status:** 🟢 **NOT VULNERABLE.**

## S6 — Stack-trace leakage to client 🟢 OK

**Risk model.** An unhandled exception inside the endpoint could surface as a 500 response whose body contains the full Python traceback (file paths, line numbers, variable names), aiding reconnaissance.

**Code review.** Uncaught exceptions inside the endpoint propagate to FastAPI's default `ServerErrorMiddleware`, which returns:

```json
{"detail": "Internal Server Error"}
```

No traceback is included in the response body. Tracebacks are only logged to the server's stderr (visible only to the operator).

The explicit `HTTPException(400, …)` calls (e.g., line 802 for missing input, line 812 for batch-size violation) return clean message strings with no path information.

**Status:** 🟢 **OK.**

## S8 — Frontend trusts server response 🟢 OK

**Risk model.** A malicious or buggy server response could crash the frontend renderer (e.g., missing fields, wrong types).

**Code review.** The renderer defends against missing fields at every read site:
- `(data.items || []).forEach(...)` (line 277) — guards against missing `items` array.
- `c = item.classifier || {}` (line 288) — guards against missing classifier block.

Furthermore, every field is rendered via `textContent` (per S1) — even if the server returned malicious markup, it would be displayed as plain text.

**Status:** 🟢 **OK.**

## S10 — Privilege escalation / auth bypass 🟦 DESIGN CHOICE

**Context.** The contribute page is **deliberately public/anonymous**. Anyone with the URL can submit a paper. The endpoint reads the optional auth context via `_get_optional_user(request)` at line 786 — if no Authorization header is present, `user = None` and the submission is recorded as `submitter_type='anonymous'`.

**Why this is correct.** The assignment specification calls for a public submission flow: the contribute page is the front door for non-students who want to suggest a paper. Requiring authentication would defeat the purpose.

**Boundaries.**
- Anonymous submissions cannot bypass dedup, magic-byte validation, or the classifier — they go through the same path as authenticated submissions.
- Anonymous submissions cannot read or modify other users' rows — the endpoint only writes new rows. There is no `GET /api/articles/{id}` exposed publicly.
- Anonymous submissions are tagged at `articles.submitter_type='anonymous'` and `articles.submitter_id=NULL` (line 769-770) so reviewers can filter / triage / rate-limit them as a class.

**Status:** 🟦 **DESIGN CHOICE.** Anonymous submission is intentional, scoped, and identifiable.

## S11 — CSRF on the submit endpoint 🟢 N/A

**Risk model.** A cross-site request forgery would matter if the endpoint relied on session cookies for auth — a malicious page on another origin could trick a logged-in user's browser into submitting an unintended request.

**Why CSRF doesn't apply here.**
- The endpoint accepts `multipart/form-data` POST without requiring or reading any session cookie.
- Authentication, if present, is via `Authorization: Bearer <token>` (read from `localStorage["ka_access_token"]`). Bearer tokens are NOT automatically sent by the browser across origins — they have to be explicitly attached by JavaScript that has access to the same `localStorage`. Cross-origin JS cannot read another origin's `localStorage`.
- Anonymous submissions are explicitly allowed (S10), so there is no escalated session to forge.

**Status:** 🟢 **N/A.**

## S12 — DB connection hygiene 🟢 ENFORCED

**Risk model.** Without explicit PRAGMAs, SQLite under concurrent writes can throw `database is locked` exceptions, allow foreign-key violations, or block the WAL checkpoint.

**Code review.** `ka_auth_server.py:227-231`:

```python
db = sqlite3.connect(str(DB_PATH), timeout=5.0)
db.row_factory = sqlite3.Row
db.execute("PRAGMA journal_mode=WAL")
db.execute("PRAGMA foreign_keys=ON")
db.execute("PRAGMA busy_timeout=5000")
```

These PRAGMAs are set **once at the shared connection setup** in the auth server and apply to every endpoint that uses the injected `_get_db()`. The `sqlite3.connect(timeout=5.0)` argument and `busy_timeout=5000` (5 seconds) together provide both library-level and engine-level wait-on-lock behavior. `journal_mode=WAL` allows concurrent readers while a writer is active. `foreign_keys=ON` ensures every FK constraint is enforced (without this, SQLite ignores them by default).

**Status:** 🟢 **ENFORCED** at the architecturally correct layer (shared connection setup, not per-endpoint duplication).

## S13 — Concurrent ID-collision attack 🟢 NOT VULNERABLE

**Risk model.** A pre-PR `_next_id` based on `SELECT COUNT(*) + 1` allowed two concurrent submissions to both compute the same ID, race into a primary-key INSERT, and crash the loser with `IntegrityError`. An attacker could deliberately submit batches to trigger 500s.

**Fix.** `_next_id` at `ka_article_endpoints.py:332-365` was replaced with an atomic counter:

```python
"UPDATE id_sequences SET counter = counter + 1 "
"WHERE prefix = ? RETURNING counter"
```

`UPDATE … RETURNING` is atomic in SQLite ≥ 3.35 (we run 3.50). Two concurrent transactions serialize on the row lock — one gets counter `n`, the other waits and gets counter `n+1`. Collision-free by construction.

**Verification.** Verified during Phase-3 Q3 testing: 10 concurrent POSTs via `xargs -P10` produced 10 distinct monotonic IDs, zero 500s. See `verification_log.md` §Q3.

**Status:** 🟢 **NOT VULNERABLE.**

## S14 — Race-condition on dedup probe 🟡 PARTIAL MITIGATION

**Risk model.** The dedup probe (`_check_duplicates` at line 505-602) runs BEFORE the INSERT. In a tiny race window between probe and INSERT, a concurrent submission of the same paper could pass dedup, then both INSERT — neither marked as duplicate.

**Mitigation in place.** Retry-on-IntegrityError loop at line 1030-1040 (per verification log Q3 fix `1cabb0a`):

```python
try:
    _do_insert(article_id)
except sqlite3.IntegrityError:
    article_id = _next_id("KA-ART")  # fresh ID
    _do_insert(article_id)            # retry once
```

If a race occurs and both submissions reach INSERT with the same `pdf_hash_sha256` and the schema has a UNIQUE constraint on that column, the second INSERT fails and is retried with a fresh ID. The retry path does not re-run dedup, however — so the second row would be created without the `duplicate_existing` status.

**Residual risk.** Two simultaneous submissions of the same paper from two clients would, in the worst case, produce two `staged_pending_review` rows instead of one `staged_pending_review` + one `duplicate_existing`. This is a degraded-but-not-broken outcome: a reviewer would catch the duplicate on review (titles/hashes are obviously identical), so no corruption occurs.

**Recommendation for production.** Wrap the probe + INSERT in an explicit `BEGIN IMMEDIATE` transaction so the probe holds a write lock until the INSERT commits. Not blocking for course submission.

**Status:** 🟡 **PARTIAL MITIGATION.** Documented; reviewer catches the worst case.
